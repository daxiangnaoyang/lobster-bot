## 认知改变型

### 示例1：SOP与Skill关系

```
我研究了100多个Skills

发现了一个让AI能力提升10倍的秘密

SOP是展开的Skill

Skill是打包的SOP

它们描述的是同一件事

只是视角不同

SOP：人类视角的步骤清单

Skill：AI视角的能力模块

这个认知能解决3个核心问题

问题1：Agent为什么"忘记"怎么做？

你只给了SOP，没有给Skill

SOP是人类视角

AI需要工具使用细节

解决：

直接提供Skill文件

包含工具调用细节

从那以后

每个重要能力

我都同时创建SOP和Skill

人类看SOP：理解完整流程

AI看Skill：掌握工具使用

错误率降低80%

效率提升10倍
```

**核心价值**：封装关系、双文件策略
**认知改变**：SOP vs Skill是对同一事物的不同封装
**行动改变**：同时创建SOP和Skill

## 问题解决型

### 示例1：Agent忘记执行

```
你的Agent为什么经常"忘记"怎么做？

明明在AGENTS.md里写得清清楚楚

明明在SOUL.md里反复强调

但Agent执行起来还是经常跑偏

我发现了3个核心问题

问题1：只给了SOP，没有给Skill

SOP是人类视角

AI需要的是工具使用细节

解决：

直接提供Skill文件

包含工具使用细节

问题2：只给了Skill，没有给SOP

Skill缺少宏观流程

AI不知道调用时机

解决：

同时提供SOP文件

说明Skill在整个流程中的位置

问题3：SOP和Skill不同步

更新了一个，忘记更新另一个

解决：

建立同步检查机制

从那以后

错误率降低80%

效率提升10倍
```

**核心价值**：Agent设计的3个核心问题及解决方案
**解决的问题**：Agent执行跑偏
**具体方案**：双文件策略 + 同步机制

## 方法论型

### 示例1：先试再问

```
一个提升 10 倍效率的方法

先试，再问

而不是

先问，再试

为什么？

因为问得到的只是答案

试得到的才是认知

举个例子

你想实现一个功能

方法 A：先问别人怎么做

→ 你得到了答案
→ 但你不知道为什么这么做
→ 下次遇到类似问题，你还要问

方法 B：先自己试

→ 你踩了坑
→ 你知道哪里会出错
→ 你知道为什么这么做
→ 下次遇到类似问题，你自己能解决

这就是区别

答案会过时

认知不会
```

**核心价值**：先试再问方法论
**方法**：对比A/B两种方法
**效果**：10倍效率提升

```
一个让AI能力提升10倍的方法

建立SOP和Skill的同步检查机制

从那以后

错误率降低80%

效率提升10倍
```

**核心价值**：双文件策略
**方法**：3个实战策略
**效果**：80%错误率降低

## 示例使用指南

### 如何选择风格

1. **有深度洞察** → 认知改变型
2. **解决具体问题** → 问题解决型
3. **有方法论** → 方法论型
4. **反常识发现** → 洞察型
5. **感谢/分享** → 价值传递型
6. **极简有力** → 极简价值型

### 开头3段检查

所有示例都在前3段点出价值：

```
第1段：[吸引注意]
第2段：[建立场景/点题]
第3段：[核心价值/洞察]
```

### 信息密度检查

好的示例：
- 每段都有信息量
- 没有情绪堆砌
- 没有废话段落

可删除的段落：
- "那一刻我意识到..."
- "说实话，我..."
- "挺意外的、挺开心的"
